B. Finish the Word: D and I

  • P knows how far is left
  • P ignores how fast it’s closing — or that a lean is stealing from it
  • Two letters fix that

Break it: P in a hurry 🔥

heading = 0°  target 90°
P sprints in… and slams the door. 🚪
  • The push only knows how far is left — not how fast it’s closing
  • It arrives at 90 flat-out, blows past, over-corrects, and rings before it settles
  • A bigger gain rings harder

Run it before the fragment. The red arc painted past the tick is the overshoot made physical — point at it. Elicit the diagnosis in their words: “the number hit 90 — why didn’t it stop?” Because the wheels were still spinning fast; P read error 0 and cut power, but momentum carried it past, then P yanked it back, and round it goes. This is exactly the D3 coast — but now P can’t two-speed its way out, because a hot gain is what won the race yesterday. Set up the need: we want something that notices the error is dropping fast and eases off early. Don’t name D yet.

Name it: D damps 🛑

def gyro_turn_pd(target, gain=4, kd=0.5):
    hub.imu.reset_heading(0)
    last_error = target
    while abs(target - hub.imu.heading()) > 1:
        error = target - hub.imu.heading()   # how FAR → P
        change = (error - last_error) / 0.005 # how FAST → D (°/s)
        last_error = error
        left.run(gain * error + kd * change) # ease off early
        right.run(-(gain * error + kd * change))
        wait(5)
    left.stop(); right.stop()
  • P reads where you ARE. D reads how fast you’re closing
  • Watch error fall 90 → 55 → 30: each tick change is negative (−35, then −25) — the gap is closing fast
  • D catches that and eases off early — it taps the brake before you arrive, so the turn glides in instead of ringing. That’s D — derivative, the second letter of PID

Anticipation is the whole idea — say it that way: P only knows the gap’s size right now; D knows the gap is shrinking and lifts off early, like braking as you near a parking spot instead of at it. Three new lines: remember last tick’s error, subtract to get the change (a speed, °/s), feed it in. Sign check out loud on the card’s numbers — as the robot closes, error shrinks each tick, so change is negative, so it trims the command down → gentle arrival. kd is a new knob, same as gain was — a strength dial for the braking. Too little kd = still rings (P alone); too much = it brakes so early it crawls in sluggishly. They’ll feel both on the next slide. gyro_turn_pd stays on run() (see the D4 proportional-turn note on run() vs motor.dc()); worth knowing for coaches — and it pays off again at the D10 reveal, where the pro controller actuates one layer deeper, by torque/effort.

The last letter: I — (coach demo) 🧲

Rest a thumb of weight on one side. Now the P-turn stops a hair short — 88°, not 90 — and stays there, forever. Why can’t P finish the job?

🧲 = the lean, riding the car

  • P’s push (gain × error) can’t beat the lean, so it parks a hair short forever — that’s yesterday’s gain = 1, now explained

This is a coach demo — never gated. Show it, don’t assign it. Fix it: a file of your own — my_lib.py 📦

# my_lib.py — YOUR moves: written once, named forever
from pybricks.hubs import PrimeHub
from pybricks.pupdevices import Motor
from pybricks.parameters import Port, Direction
from pybricks.tools import wait

hub = PrimeHub()
left = Motor(Port.A, Direction.COUNTERCLOCKWISE)
right = Motor(Port.E)

KP = 4                                # named constants — tune HERE,
WHEEL_MM = 88                         # nowhere else
DEG_PER_MM = 360 / (3.14 * WHEEL_MM)  # Day 2's wheel math, bottled
  • One new file. The top holds the robot’s facts (ports, gains, wheel math); below it live the verbs
  • Your mission file just says from my_lib import drive_straight, turn_to — and reads like a story instead of a swamp

Make the file for real, together — everyone creates my_lib.py next to their mission file. The constants block is the same discipline from the square day, now promoted to the library’s front door: facts at the top, verbs below, missions elsewhere. DEG_PER_MM bottles the D2 rope-unroll math (360° per wheel-turn ÷ circumference) so distance-in-mm is a multiply away. Two verbs move in today; the library keeps growing all week — wall verbs next session, line verbs after — same file, more names.

Verb #2: drive_straight — a straight with a finish line

def drive_straight(distance_mm, speed=400):
    """Drive distance_mm, healing the heading the whole way."""
    heading = hub.imu.heading()                    # lock THIS heading
    finish = left.angle() + distance_mm * DEG_PER_MM
    while left.angle() < finish:                   # encoder finish line
        error = hub.imu.heading() - heading        # one heartbeat
        left.run(speed - KP * error)
        right.run(speed + KP * error)
        wait(5)
    left.stop(); right.stop()
  • Day 2’s wheel math sets the finish line in degrees; Day 4’s healing loop steers every heartbeat on the way there
  • A verb calling a verb would stack — here it’s inlined for simplicity, same idea: libraries grow by stacking, not pasting

Single function now: drive_straight handles both the finish line (encoder degrees) and the healing loop (P steering each tick). finish is measured from wherever the wheel already is (no encoder reset), same absolute-not-relative discipline as turn_to. Backwards moves (a negative distance_mm) need the comparison flipped — park it as a challenge today; it becomes urgent at the wall session’s measured back-off, and the coach exemplar handles it. Benchmarks come at the gate: 1 m within ±2 cm is very reachable with a measured wheel. The speed=400 in the def line gets its name — default — on the very next slide.

Knobs with defaults 🎛️

def turn_to(target, gain=4, kd=0.5):         # two knobs, factory-set
def drive_straight(distance_mm, speed=400):  # one knob, factory-set
  • Every = in a def line is a default — a factory setting. Skip the knob, the factory value runs; name it — a keyword argument like kd=0 — and you turn that one knob, nothing else

Predict — what runs? ① turn_to(90) · ② turn_to(90, kd=0) · ③ drive_straight(300, speed=150) · ④ drive_straight(speed=150)

① all factory settings — your tuned gain=4, kd=0.5 run unasked · ② brake knob to zero — yesterday’s ringing P is back, on demand 🔔

③ one careful straight at 150 — every other call still fast · ④ 💥 TypeError — distance_mm has no default. Some knobs must be turned

Name the vocabulary plainly, once: the names in the parentheses are parameters; a parameter with = carries a default; naming a knob at the call site (kd=0) is a keyword argument. They’ve used all three since gyro_turn_p(target, gain=4) — this slide only hangs names on habits. Quiz cadence: read a call, every squad commits out loud, then reveal the card — ② is the fun one (one keyword resurrects yesterday’s ringing bug, proof the default was doing real work), ④ is the confidence move: run it live, read the TypeError together — the interpreter is telling you which knob you forgot, not scolding you. Forward hook for coaches: at the D10 reveal, RobotConfig gathers all these factory settings into one dataclass — same idea, bigger box.

Two homes for a name 🗺️

# top of my_lib.py — GLOBALS: born once, every verb reads them
hub = PrimeHub()
KP = 4

def drive_straight(distance_mm, speed=400):
    heading = hub.imu.heading()              # LOCALS: born at the call…
    error = hub.imu.heading() - heading      # …gone the moment it ends

The run just ended — what does print(error) print? And: change KP = 4 → 6 at the top — how many verbs feel it?

print(error) → NameError — error is a local: born at the call, dead at the end. A feature — every verb gets private scratch names that never collide

KP is a global — one edit, every verb feels it. The file’s whole shape: facts up top (global) · verbs below · scratch inside (local)

Two words, two homes — point at the actual file while saying it: a name born at the top of the file is a global (born once, when the file loads; every verb below can read it — hub, left, KP, DEG_PER_MM); a name born inside a def is a local (born fresh at every call, gone at the end — heading, finish, error). Run the print(error) NameError live if a laptop’s handy; kids expect the number to still be there, and the error landing is the lesson. Then sell dying-locals as a feature: turn_to and drive_straight both use error and never trip over each other, and every call starts with a clean slate. The KP question is the payoff of the “tune HERE, nowhere else” comment they already wrote — one global, read by everyone, edited in one place. Don’t mention the global keyword; the next slide handles the one trap without it.

Break it: the tune that didn’t stick 🔥

def make_it_sharper():
    KP = 12          # crank steering for the whole library… right?

make_it_sharper()
drive_straight(300)  # …still lazy. Nothing changed. 😤
  • Assigning a name inside a def births a brand-new local — even when a global already owns that name. This KP lived inside make_it_sharper… and died at the end of the call. The top-of-file KP never moved
  • The fix is zero new code — it’s the rule the file already taught you: read a global from anywhere; change it only at the top. KP = 12 belongs in exactly ONE place — the constants block

Act it out before explaining: type make_it_sharper live, run the straight, watch nothing change — let the 😤 be theirs. Then the one-sentence rule: the moment you assign a name inside a def, Python births a new local, even if a global already wears that name — the global is untouched, and the local evaporates at return. (Inside make_it_sharper the new KP even hides the global — same name, different box.) Do not teach the global keyword as the escape hatch — the library discipline is strictly better and it’s already on the slide: constants change at the top, nowhere else. Coach’s radar for Weeks 2–3: when a squad swears “I tuned it and it didn’t stick,” the first question is “where did you assign it?” — this slide is that bug, met safely in advance.
