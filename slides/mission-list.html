B. The Plan That Couldn’t Change

Your verbs work. Now the judge calls a sequence — and then changes it. 😈

Unplugged: the pigeonholes 📮

Six numbered boxes on the bench, a pile of colored bricks. “Red into box 0. Blue into box 2. What’s in box 1? Swap 0 and 3. Add a green one at the end — which box did it land in?”

You just used every list skill: a numbered row of slots, filled in order, counting from 0 — read a slot by its number (index), walk the row front to back (iterate), grow it at the end (append). Python calls this row a list. No laptop needed — you already think this way.

Run it physically before any code — 5 minutes, real boxes (or taped squares) and real bricks, quick-fire calls. The two stumbles to engineer on purpose: (1) “box 1” when they expect counting from 1 — let someone grab the wrong brick, then name the rule: robots count from 0; (2) append lands at the end, position = old length. Keep the boxes on the bench all day — every list question later gets answered by pointing at them. This is WRO-Learn’s arrays intro compressed into a game.

Break it: the pasted plan 🔥

The judge calls: cross the stage · face the band · square on the wall. You paste it as one block. Then: “Actually — wall square FIRST.” How long does the edit take?

# mission_friday.py — one pasted block 😬
drive_straight(400)          # cross the stage
turn_to(90)           # face the band
drive_to_wall()
wall_square()         # reset at the wall
# judge: "wall square FIRST, then cross"…
# select lines… which? cut… paste… 🐛

The plan is trapped in the code. Reordering means surgery — select, cut, paste, and pray you moved all the lines of each step. Under a 5-minute clock, surgery loses. The plan needs to be a thing you can shuffle — like cards in a deck.

Let a squad actually attempt the live edit under a visible countdown — the fumble is the lesson. Diagnose together: the steps are clear in their heads (“cross”, “face”, “square”) but the code has no such boundaries; a step is 1–3 loose lines with nothing holding them together. Two missing ideas, teased in order: give each step a name (they own def — this is a two-minute step), then put the names in a row that can be shuffled — the pigeonholes from the warm-up. Don’t show the fix yet.

C. The Mission Is a List

Name the steps. Put the names in a row. Play the row.

Lists in sixty seconds

setlist = ["intro", "solo", "finale"]   # a numbered row of slots
setlist[0]                # "intro" — count from 0, like the boxes!
setlist.append("encore")  # grows at the end → slot 3
setlist[1] = "duet"       # swap what a slot holds
for song in setlist:      # visit every slot, front to back
    print(song)

Index · append · iterate — the three moves from the pigeonhole game, now in Python. A list holds anything: strings, numbers… and (next slide) something sneakier. 😏

Type it live in the REPL, mirroring each line back to the physical boxes on the bench (“slot 0… that box”). Off-by-one and negative index (setlist[-1] = last box) are fun 30-second asides if the room is quick. Don’t linger — the payoff slide is next, and lists only get interesting once functions go inside them.

Steps are just functions — hold the name, don’t play it

def cross_stage():
    drive_straight(400)

def face_band():
    turn_to(90)

def reset_at_wall():
    drive_to_wall(); wall_square()

mission = [reset_at_wall, cross_stage, face_band]
#          ^ no parentheses — cards in a deck, not moves happening

cross_stage() plays the move. cross_stage — no parentheses — is the recipe card itself, a thing you can hold, hand around… and drop into a list. A list of names is a setlist: the whole show, and it hasn’t started yet.

This is the day’s one genuinely new idea — a function is a value. Land it with the recipe-card metaphor from the first def day: the card vs cooking the dish. Common bug to demo on purpose: write mission = [cross_stage(), face_band()] — the robot drives while the list is being built (the moves happen at line 10!) and the list ends up holding three Nones. Let them see that, then delete the parentheses. Note the judge’s reorder is now already solved on line 10 — reset_at_wall simply moved to slot 0.

The runner — the whole thing

def run_mission(steps):
    for step in steps:   # take the next card…
        step()           # …now play it
run_mission(mission)

Three lines, and they never change again. Reorder the show → edit the list. Insert a step → append or slot it in. The judge changes the mission; you change data, not code. That’s the split the pros live by.

Read it aloud in pigeonhole language: for each box front to back, take the card out, play it. The deep point — worth saying explicitly, once: the runner is control and the mission is data, and everything that changes lives in the data. This exact runner drives the anchor run in Week 3; per-step timeouts and a transition log are its stretch upgrade (the exemplar’s run_mission_logged) — offer to fast squads at the gate, never gate on it. A non-blocking state machine is continuation-season material; RoboMission runs are sequential, blocking is right.

D. Compete & Gate

Today’s showdown: Setlist 🎵

Setlist 🎵

The challenge

The judge calls a 3-step mini-mission — you have 5 minutes to encode it as a mission list over your own library and run it. Score = steps completed; tiebreak = time.

  • Hear the mission → say it as verbs (“that’s a drive_straight, a turn_to, a wall_square”) → wrap, list, run.
  • The judge will change the order on round two. 😈 Your surgery-free edit is the whole advantage.
  • Fast squads: swap in run_mission_logged — per-step stopwatch + a printed transition log.

Score decomposition, not typing speed: the squads to praise are the ones who translate the judge’s sentence into library verbs before touching a keyboard. Round two’s reorder is the day’s thesis under pressure — a list squad edits one line; watch for anyone still cutting and pasting blocks and coach them on the spot. The logged runner is the exemplar’s stretch (timeouts + transition log) — offer it, never gate it. Coach’s private scorecard today: list fluency, decomposition into verbs, debugging a sequence (which step failed? the log says).
